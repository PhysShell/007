# Consilium MCP Layer — Assessment for 007

Source of record: `core/src/mcp.rs` (module doc `mcp.rs:1-20`), tests in `core/tests/mcp_test.rs`, shared confinement in `core/src/confine.rs`, spawn invariant in `core/src/sessions.rs`.

The MCP server runs in **attached-conductor mode over stdio** (`mcp.rs:796-804`, `serve_stdio`). Design intent is explicit: the conductor is the user's live interactive Claude Code session, and this server exposes Consilium's worker/quota primitives as tools so that session can orchestrate the "army" (Codex/Gemini/Grok/fallback Claude) without spending programmatic Claude credit (`mcp.rs:1-8`). It is a thin, stateless tool facade over existing library functions.

## Invariant checks

- **cwd confinement — PASS (for the tools that take a cwd).** `run_worker`, `review_diff`, `council_run` all call `crate::confine::cwd_within_root` before running, refusing uniformly (`mcp.rs:467-473,478-492,584-600,665-679`). `cwd_within_root` canonicalizes both sides so `..`/symlink escapes resolve first, and rejects a non-existent path (`confine.rs:14-19`; tests `confine.rs:39-78`, `mcp_test.rs:610-692`). Rationale stated: the conductor is an LLM steered by untrusted repo content, so `cwd` is untrusted input (`mcp.rs:16-20,478-481`). `search_recall`/`page_in` read the transcript store, not a working tree, so are not cwd-confined.
- **write/advisory invariants — PASS, structurally enforced.** `run_worker` never exposes an `advisory` param, always `advisory:false, write:true` (`mcp.rs:516-531`); read tools always `advisory:true, write:false`. Underneath, `sessions.rs:84-88` hard-bails (not a `debug_assert`, holds in release) if `advisory && write` are both set — the "most permissive, least protected" combination is impossible at the spawn boundary. Test `mcp_test.rs:140-149`.
- **Typed request/response — PASS.** Every tool has `#[derive(Deserialize, JsonSchema)]` params + `#[derive(Serialize, JsonSchema)]` output (`mcp.rs:84-246`), via `rmcp`'s `Parameters<_>`/`Json<_>`; schemas from `schemars`.
- **Failure handling — PASS.** Every path returns structured `ok:false` + `error: Some(...)`, never panics (`mcp.rs:117-118,187-188,533-545,641-649,714-720`). Reviewer fail-closed: `parse_ok:false` must be treated as unusable even when `ok:true` (`mcp.rs:145-149,331-339`; test `mcp_test.rs:459-486`).
- **Idempotency — FAIL (absent).** No idempotency key, no de-dup. Calling `run_worker` twice re-runs the worker, re-spends quota, re-writes files. Acceptable for an interactive conductor, hazardous for an automated caller.
- **Cancellation — FAIL (absent).** No cancel at the MCP boundary; a tool call runs to completion or hits its per-attempt timeout. Contrast the WS server (`{"kind":"cancel"}`). Only way to stop is killing the stdio transport.
- **Timeout — PASS.** Per-attempt 600s (worker) / 900s (review, council), threaded into `run_with_failover`/`run_review_ladder`/`run_council` (`mcp.rs:44-46,512,601,680`).
- **Duplicate / concurrent tool calls — SAFE but not deduped.** Handlers stateless; the only shared mutable state, `QuotaStore`, serializes on its own mutex (`mcp.rs:64-66`). But no single-run semaphore like the WS server — two concurrent `run_worker` into the same cwd would interleave git diffs (WS guards this; MCP does not).
- **Transcript size caps — PARTIAL.** `page_in` digests capped at `PAGE_IN_DIGEST_CAP = 4000` (`mcp.rs:47-49,758-763`); `search_recall` defaults to 10 hits (`mcp.rs:399`). **But `run_worker`'s `worker_report` and `changes` (full diff) are returned uncapped** (`mcp.rs:560-567`) — a large diff floods the conductor context. Only the recall path is bounded.
- **Prompt-injection implications — PARTIALLY mitigated.** cwd confinement is solid: an injected `cwd` cannot point a write-enabled worker or `run_verify` commands outside the launch root (`mcp.rs:16-20`; tests `mcp_test.rs:586-668`). But tool *outputs* (`worker_report`, `changes`, `raw_review`, council answers) flow back into the conductor context unfiltered, so repo-borne instructions can still attempt to steer the conductor's *decisions* — confinement bounds where writes land, not what the conductor is told. No output sanitization layer.

## The five MCP answers

**1. Suitable as a TEMPORARY bridge Claude → Consilium workers?** Yes — literally what it was built for (`mcp.rs:1-8`); clean, well-tested: typed contracts, hardcoded safe flags, cwd confinement, structured non-panicking errors, per-attempt timeouts. As a stopgap letting a Claude session drive Codex/Gemini/Grok workers at zero programmatic Claude cost, it is production-grade. Temporary-only caveats: stdio single-client transport, no run identity, no cancellation, no idempotency, no live-progress persistence, uncapped worker output.

**2. Which tools could be adapted into 007?** `run_worker` (core delegation primitive; typed params + `advisory:false/write:true` invariant is the shape 007 needs). `review_diff`, `council_run` (read-only second-opinion/deliberation; keep the fail-closed `parse_ok`/`has_critical` contract verbatim). `quota_status` (routing input, but 007 should read from o7d's own store). `search_recall`/`page_in` (adaptable only as **read views over o7d's ledger**, not their own filesystem transcript store; their query/digest ergonomics are a good UX template, backing store must be replaced).

**3. Why must MCP NOT become the source of truth?** It holds none. Stateless per-call over stdio (`mcp.rs:796-804`); owns no run identity, no ledger, no cursor. Live worker progress is not persisted by this layer — it returns a final blob. The transcripts `search_recall`/`page_in` read are written **by the CLI command paths in `main.rs` (`main.rs:447-458,514-525,584-630`), not by the MCP server** — so MCP's "recall" reads state some *other* process produced, and `page_in` is deliberately lossy/size-capped (`mcp.rs:758-763`). A source of truth needs durable, ordered, per-event, run-identified state with a cursor; MCP is a fan-out of library calls with no memory that dies with its client.

**4. Where does 007 need a capability token or run-bound socket instead?** Everywhere MCP substitutes process trust + cwd confinement for authorization + run binding: (a) No caller identity/authz — MCP trusts whoever holds the stdio pipe; once network-exposed and multi-conversation, each delegation needs a **capability token** scoping one run (cwd/root, permission set, expiry). (b) No run binding — a tool call is anonymous, no run id to cancel/resume/re-attach; 007 needs a **run-bound socket/handle** (token names a run in o7d; transport only *subscribes*). (c) No cancellation channel — a capability/run handle is where cancel/pause/resume/interject authorization would live.

**5. Can Consilium MCP schemas be the basis for 007 typed delegation?** Yes, as a **template**, not as-is. The `serde` + `schemars::JsonSchema` typed-contract pattern (`mcp.rs:84-246`) is exactly right; `RunWorkerParams/Output`, `ReviewDiffParams/Output`, `CouncilRunParams/Output` are clean fail-closed delegation contracts. To become 007's delegation layer they must be **extended** with: `run_id`, capability token, idempotency key, and (outputs) a monotonic `seq`/cursor + size caps on `worker_report`/`changes`. Keep the shapes and invariants; add identity, idempotency, ordering.
