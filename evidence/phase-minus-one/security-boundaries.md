## The 5 categories — which layer owns each

**(a) Git / worktree lifecycle safety** — `safety/git.rs`, `safety/preflight.rs`. Owns: hooks, checkout filters, apply gating (`source_is_applyable`), worktree isolation. **Status: built + heavily tested as a library, but NOT active in any run path.** `create_detached_worktree`, `remove_worktree`, `reopen_prepared_worktree`, `source_is_applyable` have zero callers outside `safety/mod.rs` re-exports and `core/tests/`. `SafeWorktree` appears only as a *reported* preflight mode (`preflight.rs:135-137`); nothing consumes it to actually isolate a run. Every real run executes **in-place** in the cwd (`main.rs` Conduct/Auto use `std::env::current_dir()`; server uses `launch_root`).

**(b) Process sandboxing (Sandboy)** — `sessions.rs:76-103` + adapter flags. Owns: secret reads, network egress, process spawn, Docker socket, npm scripts, out-of-cwd writes. **Status: ABSENT at the Consilium layer.** No seccomp/namespace/container; child inherits full env; the only "sandbox" is each provider CLI's own flag, and they are inconsistent: codex `--sandbox workspace-write` (`codex.rs:26`), claude `--permission-mode acceptEdits` (`claude.rs:31`), **gemini `--dangerously-skip-permissions` = no sandbox** (`gemini.rs:53`). Exactly where the 007 invariant "Sandboy is THE process boundary (worktree is not a sandbox)" is unmet — and the worktree isn't even active (category a), so today there is *neither* a worktree *nor* a sandbox around agent writes.

**(c) Network policy** — `server.rs` (`origin_allowed`, loopback bind) + `git.rs` env stripping. Owns: inbound UI origin (RT, well-tested) and git's SSH/askpass surface (RT). **Status: inbound loopback solid; outbound egress from agent/verify processes is unrestricted (absent).**

**(d) Agent policy** — `mcp.rs` tool invariants (`advisory:false/write:true` never a parameter, `mcp.rs:516-526`; cwd confine on every tool), conduct gates. Owns: prompt-injection blast radius, model policy, DONE≠acceptance. **Status: partial** — confinement is real+tested; there is no model-lock; and the acceptance verdict is produced by LLM roles, not an independent authority.

**(e) Evidence verification** — `orchestrator/verify.rs`, `safety/commands.rs`, `safety/trust.rs`. Owns: verifier trust and false-completion. **Status: provenance tracked but the trust store is unwired, and the verdict is LLM-owned.**

## Prose summary — the biggest gaps 007 must own

1. **No exact-lock / kill-switch, and model switching is automatic and ungated.** `run_with_failover` (`resilience.rs:190-317`) silently demotes to a different `(provider, model)` on unavailable/rate-limited/transient/spawn-error, and `cross_family_ladder` (`conduct.rs:1237`) injects models a role was never configured for. Nothing requires operator consent. 007's `ModelPolicy::Exact` must be enforced *at this choke point* (halt on switch, forbid cross-family reorder).

2. **Sandboy is the missing process boundary — and the worktree that was supposed to substitute for it is not even wired in.** The safe-worktree machinery in `git.rs` (thorough, tested) has no production caller; runs happen in-place. No OS-level sandbox, no env scrubbing (`sessions.rs:94-103`), and gemini runs with `--dangerously-skip-permissions`. Secret reads, network exfiltration, arbitrary process spawn, Docker socket, and npm lifecycle scripts all fall through to the host. Single largest surface, squarely Sandboy's to own.

3. **Untrusted repo config supplies both verify commands and model config, with the trust gate disconnected.** `Config::load` reads `consilium.config.json` from the cwd (`config.rs:231`), and `run_verify` executes resolved commands via `sh -c` (`verify.rs:75-99`) with **no** consultation of the `TrustStore` — which exists and is tested but has no production caller. A hostile repo therefore achieves RCE-on-verify and can redirect the whole council's models. 007's "trusted verification commands must not come from untrusted repo config" requires wiring trust-on-first-use into the verify path and sourcing model/verify config from a trusted location, not the target repo.

4. **The verdict is LLM-owned, not o7d-owned, and several gates fail open.** Acceptance rests on LLM evaluator/arbiter output (`conduct.rs:657-680,1188-1214`); verify can be silently `not_run`; the reviewer-unavailable path *accepts* (`conduct.rs:1102-1108`); and quota reads fail open to "zero usage" (`server.rs:231`, `mcp.rs:448`). For 007, only o7d may own the verdict, "DONE" must never imply acceptance, and each of these must be made fail-closed.
