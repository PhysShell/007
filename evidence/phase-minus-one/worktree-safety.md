# worktree-safety.md

## Intro

Read-only audit of Consilium's worktree-safety subsystem, which lives entirely under `core/src/safety/` (`git.rs`, `fs.rs`, `trust.rs`, `commands.rs`, `preflight.rs`, `mod.rs`) plus the shared path helper `core/src/confine.rs`. Studied against the two dedicated integration suites (`core/tests/worktree_test.rs`, `core/tests/trust_test.rs`) and the in-module unit suite `core/src/safety/git.rs:1528-1838`. No files were modified and no tests were run by this study.

What the subsystem actually is: a **filesystem-and-Git-content isolation boundary**. `create_detached_worktree` materializes a repository's committed `HEAD` into an owner-only directory tree outside the source repo, hands back a live, descriptor-bound cleanup capability, and guarantees that both creation and teardown fail closed (preserve everything) if it cannot prove — via device/inode identity re-checks against held file descriptors — that it is still operating on the exact directories it created. It is explicitly **not** a process boundary: it constrains where committed bytes land and how they are torn down, not what a process running inside may do. Per 007 invariant #10 this document treats a worktree as an isolation directory, never as a sandbox.

The single most load-bearing architectural finding: **none of the worktree-lifecycle API is wired into shipping code.** The only production consumer of the whole `safety` module is `core/src/orchestrator/verify.rs:2`, which imports `resolve_commands_with_provenance` and `VerificationCommand`. `create_detached_worktree`, `remove_worktree`, `reopen_prepared_worktree`, `TrustStore`, and `preflight::inspect` are exercised **only by tests** (grep of `core/src` for each symbol returns zero non-test, non-`safety/` hits). The machinery is real, hardened, and heavily tested — but currently dormant.

"PROVEN" means a test asserts the behavior; "COMMENT/CODE-ONLY" means the property rests on the implementation and its doc-comments with no test that specifically exercises it.

## The 8 answers

### 1. Which threats does this code ACTUALLY close?
Edit leakage into the operator's checkout (`worktree_test.rs:21-39,58-88`); repository-controlled code execution during setup — hooks, smudge/process filters, fsmonitor, external-diff, `GIT_SSH_COMMAND`, config-injection env (`git.rs:245-274,1418-1456`; `worktree_test.rs:461-593`); TOCTOU path substitution of state root / worktrees parent / entry via symlink/rename/inode swap, failing closed and preserving files (`git.rs:1577-1689,1751-1792`); destructive cleanup escaping its bounds — source-repo deletion, symlink escape, unrelated-worktree force-remove, cross-device (`worktree_test.rs:300-315`; `git.rs:1657-1689`); state root placed inside/symlinked-into the source repo (`worktree_test.rs:234-298`); forged/replayed cleanup authority after restart (`worktree_test.rs:317-398`); world-readable secrets/state (`trust_test.rs:52-72`; `worktree_test.rs:400-420`); applying unreviewed operator drift (`worktree_test.rs:97-121`).

### 2. Which claims are proven by tests?
Everything marked PROVEN. Strongest: the phase-hook substitution probes in `git.rs`'s unit suite (`git.rs:1577-1837`) inject symlink/rename/inode swaps at precise `BeforeGit`/`AfterGit`/`BeforeMissingEntryPrune`/per-child-quarantine moments and assert (a) "identity changed"/"preserving" error, (b) both attacker+victim files survive, (c) source repo stays `clean`. Cross-process reopen (`worktree_test.rs:332-375`) is a genuine second-process boundary. Trust-store fd-binding under parent substitution proven at `trust_test.rs:110-133`.

### 3. Which claims rest only on comments (unproven)?
Cross-filesystem-device refusal (`git.rs:398-400,1224-1241`) — code only, no second-fs test. Concurrent-creation safety (`git.rs:745-785`, `fs.rs:126-145`) — by construction, no contention test. Live-remove marker/token *swap* (`validate_marker` `git.rs:1021-1030`) — untested (only reopen-time match tested). `PreparedWorktree` has **no `Drop`** cleanup (only `CreationReservation` does, `git.rs:331-338`) — a dropped *successful* handle leaks the worktree; the RAII-vs-explicit contract is comment-only. The `GIT_*` env-scrub list's exhaustiveness (`git.rs:1434-1453`) — by construction; tests cover only hooks/filters/fsmonitor.

### 4. What remains OUTSIDE the boundary (explicitly not handled)?
All process containment (no seccomp/namespace/uid/network isolation): `run_git` (`git.rs:1418-1456`) and `verify.rs`'s `sh -c` with `.current_dir(cwd)` (`orchestrator/verify.rs:96-105`) run with the host's full authority; a process in the worktree can read/write outside it via absolute paths, open sockets, exec anything. Windows/non-Unix (fails closed, WSL only, `git.rs:203-210`). Repo-*local* `.git/config` reads (mitigated by never checking out). The trust store's model is command-tampering, not command confidentiality. `cwd_within_root` (`confine.rs`) is a weaker canonicalize-then-`starts_with` check (subject to a post-check swap the worktree code is not). Verify-command budgeting lives in `verify.rs`, not here.

### 5. Could the module be extracted into a standalone crate? What would it drag along?
Yes, in two tiers. **Tier 1 = `git.rs`+`fs.rs`** is nearly free-standing: only `anyhow`, `serde`, `rand`, `rustix`(unix), `std` (`core/Cargo.toml:10,16,20-23,30`); zero `crate::` imports. The only coupling to sever is `ts-rs` `#[ts(export_to="../../ui/...")]` on inspection DTOs `RepositoryKind`/`RepositoryState` (`git.rs:19-38`). **Tier 2 = trust+preflight+commands** does NOT extract cleanly: `commands.rs` pulls `crate::config::VerifyConfig` + `crate::orchestrator::verify` (`commands.rs:1-2`); `preflight.rs` pulls `crate::config`, `crate::protocol::ConfigSummary`, `crate::orchestrator::verify`, `crate::confine` (`preflight.rs:1-7`). Recommended: ship `git.rs`+`fs.rs` as `worktree-isolation`; leave trust/preflight/commands in Consilium.

### 6. Which APIs would have to change for 007?
`create_detached_worktree(source, state_root)` (`git.rs:203`): caller chooses `state_root`; for o7d single ownership it must be an o7d-owned anchor + associated with an o7d run id (no run/worktree linkage exists today). Ownership/`Drop`: add RAII cleanup or make the persist-summary-then-remove/reopen contract mandatory+typed (today a forgotten handle silently leaks). Blocking API: all git calls are synchronous `std::process::Command` (`git.rs:1418`) → wrap in `spawn_blocking`, add cancellation token. Persistence: `PreparedWorktreeSummary` (`git.rs:63-69`) + state root + `reopen_prepared_worktree` (`git.rs:372`) is already the right shape for a single persistent owner — keep it, o7d owns the summary store. Remove/parameterize the `ts-rs` UI export path. Non-Unix: return a typed unsupported-platform variant instead of a string error.

### 7. How does it interact with Sandboy (the process boundary)?
It does not, and must not be confused with one. Zero process isolation: `run_git` (`git.rs:1418-1456`) and `verify.rs`'s `sh -c` (`orchestrator/verify.rs:96-105`) are ordinary child processes with the host's full authority. **Textually Consilium does NOT conflate them**: "sandbox" in `core/src` appears only in adapter flags passed to the agent CLIs' own sandboxes (`adapters/codex.rs:26` `--sandbox workspace-write`; `adapters/grok.rs:29`; `adapters/mod.rs:22`), never on the worktree. The real risk is the `ExecutionMode::SafeWorktree` label (`preflight.rs:17-21`): "Safe" can be misread as "sandboxed." For 007: keep "worktree" strictly for the filesystem boundary; let Sandboy be the sole process boundary. This subsystem is the correct thing to hand Sandboy as the writable working directory — it is not, and does not claim to be, the sandbox.

### 8. Can it be reused WITHOUT creating two competing owners of the worktree lifecycle?
Yes — unusually clean, because Consilium has **no live owner today** (only `verify.rs` uses the module, for command resolution; `conduct.rs:330-332` explicitly *defers* worktree-per-subtask isolation). o7d can be the sole caller of create/remove/reopen from day one. Two facts to respect: (1) the authority model is already single-owner-shaped — live authority is a non-clonable, non-deserializable capability (`git.rs:51-57`); persistent authority is `state_root` + `PreparedWorktreeSummary` reconstituted only via `reopen_prepared_worktree` (`git.rs:372-447`); if o7d owns the state root + summary store, no other process can fabricate deletion authority (`worktree_test.rs:377-398`). Hand Consilium *paths* produced by o7d, never let it call `create_detached_worktree` itself. (2) `git worktree prune --expire now` is **repo-global** (`git.rs:1118-1122,1163-1167`); two independent owners would race repo-wide prunes → single pruner required. o7d as sole owner eliminates the race by construction.

## Threats closed vs out-of-boundary (summary)
**Closed (test-proven except Q3 exceptions):** edit leakage into operator checkout; repo-controlled code execution via hooks/filters/fsmonitor/config-env during setup+inspection; TOCTOU symlink/rename/inode substitution of every state directory during creation+deletion; destructive cleanup escaping via symlink/unrelated-worktree/source-repo deletion; state root inside/symlinked-into source; forged/replayed authority after restart; world-readable state; applying unreviewed operator drift.
**Out of boundary:** all process containment (Sandboy's job, invariant #10); native Windows/non-Unix (fails closed, WSL only); repo-local `.git/config` reads; cross-fs-device refusal + concurrent-creation (code-only, untested); the weaker `cwd_within_root`; verify-command budgeting; and the overarching gap — the entire lifecycle is currently **dormant**, so none of these guarantees protect any shipping run today.

## Extraction & 007 integration
**Crate boundary:** Tier 1 = `git.rs`+`fs.rs` → standalone `worktree-isolation` (self-contained; sever the `#[ts(export_to=...)]` on `RepositoryKind`/`RepositoryState` `git.rs:19-38`, feature-gate or drop `inspect_repository`). Tier 2 = `trust.rs`+`preflight.rs`+`commands.rs` stays with Consilium.
**API changes:** explicit ownership (run-id + RAII `Drop` or mandatory persist-summary contract, since a dropped success handle leaks — `git.rs:51-59`); async wrappers with cancellation; typed unsupported-platform result; parameterized/removed UI export path; keep `state_root`+summary+`reopen` as the persistence API.
**Sandboy relationship:** this crate produces the writable working directory Sandboy is permitted to touch, never the isolation mechanism; they compose, neither substitutes. Avoid re-exporting "SafeWorktree" naming (`preflight.rs:18`).
**Single-owner concern:** low risk to adopt (no incumbent owner). Make o7d sole owner of `state_root`, the summary store, and every create/remove/reopen/prune for a given repo. The non-deserializable capability and the repo-global prune both force single ownership; o7d as source of truth eliminates the prune race by construction.
